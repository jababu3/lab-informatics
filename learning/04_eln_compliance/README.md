# Module 04: Electronic Lab Notebooks & 21 CFR Part 11

## The Role of the Lab Notebook

The laboratory notebook is the primary record of scientific work. It documents what was done, by whom, when, and what was observed. In regulated contexts — FDA inspections, patent disputes, IP litigation — the notebook is often the central piece of evidence.

Paper notebooks are vulnerable to physical loss, degradation, and undetected forgery. Electronic Lab Notebooks (ELNs) solve most of those problems but introduce new ones: digital records can be silently edited, backdated, or duplicated without leaving physical traces.

**21 CFR Part 11** is the FDA regulation that defines the criteria under which electronic records and electronic signatures are considered equivalent in trustworthiness to paper records and handwritten signatures.

This module explains the key provisions of Part 11 and how this project models them. The implementation is educational — it demonstrates the concepts but does not constitute a validated, production-grade compliance system. The gaps are called out explicitly below.

---

## Key Provisions of §11

### §11.10 — Controls for Closed Systems

This section applies to systems where access is controlled by the organization operating them — which describes our login-protected ELN.

**§11.10(a) — Record Integrity**
> "Validation of systems to ensure accuracy, reliability, consistent intended performance, and the ability to discern invalid or altered records."

**Implementation:** Each ELN entry receives a SHA-256 hash computed from its content at creation time. A second hash is computed at signature time. If the two hashes differ, the record was modified between creation and signing.

```python
def _content_hash(data: dict) -> str:
    serialized = json.dumps(data, sort_keys=True, default=str)
    return hashlib.sha256(serialized.encode()).hexdigest()
```

If someone bypasses the application and directly modifies the MongoDB document, the stored hash will no longer match the recomputed hash, making the tampering detectable.

**§11.10(c) — Record Protection**
> "Protection of records to enable their accurate and ready retrieval throughout the records retention period."

**Implementation:** Once an entry is signed, the application blocks all further modifications:

```python
if entry.get("status") == "signed":
    raise HTTPException(status_code=403, detail="Cannot modify a signed record")
```

**§11.10(e) — Audit Trails**
> "Use of secure, computer-generated, time-stamped audit trails to independently record the date and time of operator entries and actions that create, modify, or delete electronic records."

**Implementation:** Every state-changing operation appends an event to an `audit_log` array embedded in the document:

```python
def _audit_event(action: str, actor: str, detail: str = "") -> dict:
    return {
        "action": action,
        "actor": actor,
        "timestamp": datetime.now(timezone.utc).isoformat(),
        "detail": detail,
    }
```

Example audit log:
```json
[
  {"action": "created",           "actor": "jsmith", "timestamp": "2026-03-28T14:00:00Z"},
  {"action": "document_uploaded", "actor": "system", "timestamp": "2026-03-28T14:05:00Z"},
  {"action": "signed",            "actor": "jsmith", "timestamp": "2026-03-28T14:30:00Z"}
]
```

**Trade-off:** A production-grade audit trail would record *what changed* (old value → new value), not just *that something happened*. Our implementation logs event types but not field-level diffs, which would be insufficient for an FDA audit.

### §11.50 — Signature Metadata

Each electronic signature must include:

| Requirement | §11.50 Reference | Implementation |
|---|---|---|
| Printed name of the signer | §11.50(a)(1) | `signer_name`, validated against the JWT identity |
| Date and time of signing | §11.50(a)(2) | `signed_at`, generated server-side in UTC |
| Meaning of the signature | §11.50(a)(3) | `meaning` field (e.g., "Author", "Reviewer") |

The timestamp is generated by the server, not the client. If the client supplied the timestamp, a user could backdate a signature by changing their system clock.

---

### §11.200 — Electronic Signature Components

Non-biometric electronic signatures must use at least two distinct identification components (§11.200(a)) — typically a username and a password.

**Previous implementation:** The system accepted a typed acknowledgment phrase as the signature mechanism. This provided no identity assurance — anyone who knew the phrase could sign.

**Current implementation:** Signing requires a valid JWT, which proves the user previously authenticated with their credentials. The server then verifies that the JWT's username matches the claimed signer name:

```python
@router.post("/{entry_id}/sign")
async def sign_eln_entry(
    entry_id: str,
    sig: SignatureRequest,
    current_user: User = Depends(get_current_user),
):
    if current_user.username.lower() != sig.signer_name.lower():
        raise HTTPException(403, "Token username does not match signer_name")
```

## Cryptographic Hashing

SHA-256 is a one-way hash function with two properties relevant to record integrity:

1. **Deterministic:** The same input always produces the same 256-bit output.
2. **Avalanche effect:** A minimal change to the input (even a single character) produces a completely different hash.

```python
import hashlib, json

data = {"title": "Kinase Assay", "ic50": 0.23}
h1 = hashlib.sha256(json.dumps(data, sort_keys=True).encode()).hexdigest()
# "3a7bd3e2360a3d29eea436fcfb7e44c735d117c42d1c1835420b6b9942dd4f1b"

data["ic50"] = 0.24  # one digit changed
h2 = hashlib.sha256(json.dumps(data, sort_keys=True).encode()).hexdigest()
# "9d4e1e23bd5b727046a9e3b4b7db57bd..." — completely different
```

The ELN stores a `content_hash` at creation and a `record_hash_at_signing` at signature time. If they match, the record was not modified between those two events.

Hashing is not encryption. The hash is not secret; its value lies in the fact that it is computationally infeasible to alter the data while preserving the original hash.

## The Signature Block

When a document is signed, the following structure is appended:

```json
{
  "signer_name": "jsmith",
  "signer_title": "Research Scientist I",
  "meaning": "Author approval",
  "signed_at": "2026-03-28T14:30:00+00:00",
  "record_hash_at_signing": "3a7bd3e2360a3d29...",
  "auth_method": "jwt_bearer"
}
```

The `record_hash_at_signing` cryptographically binds the signature to the exact state of the document at the moment it was signed. Any subsequent modification to the document body would produce a different hash, proving the data was altered after signing.

## Document Lifecycle

ELN entries follow a unidirectional state machine. Reverting from a signed state is not permitted.

```
[draft] ──────────────► [in_review] ──────────────► [signed]
   │                         │                          │
   │  POST /eln/        (optional)                POST /eln/{id}/sign
   │                                              (requires JWT identity match)
   │                         │
   └──── edits allowed ──────┘              edits blocked after signature
```

## What This System Does Not Implement

This project demonstrates Part 11 concepts but is not a compliant system. A production deployment would additionally require:

| Requirement | Status | Notes |
|---|---|---|
| Formal system validation (IQ/OQ/PQ) | ❌ | Requires documented test protocols and evidence packages |
| Backup and disaster recovery | ❌ | Production would need automated, geographically distributed backups |
| Standard Operating Procedures (SOPs) | ❌ | Technical controls must be paired with written administrative policies |
| Training records | ❌ | Systems must track which users completed training and when |
| Field-level audit diffs | ❌ | Our audit log records event types but not old/new field values |
| Biometric signatures | ❌ | We use token-based (non-biometric) authentication |
| Content hash at creation | ✅ | SHA-256 hash computed on record creation |
| Hash-linked signatures | ✅ | SHA-256 hash recomputed at signing and stored with the signature |
| Server-side timestamps | ✅ | All timestamps generated in UTC on the server |
| Immutable audit log | ✅ | Events appended to an array; no deletion mechanism exposed |

### Exercise 4.2: Observe the Avalanche Effect

Run in a Python shell:

```python
import hashlib, json

entry = {
    "title": "Kinase Assay",
    "author": "jsmith",
    "sections": [{"content": "Staurosporine was tested at 8 concentrations."}]
}

def content_hash(data):
    return hashlib.sha256(json.dumps(data, sort_keys=True, default=str).encode()).hexdigest()

h1 = content_hash(entry)
print(f"Original:  {h1}")

entry["title"] = "Kinase Assay "  # one trailing space added
h2 = content_hash(entry)
print(f"Modified:  {h2}")
print(f"Match: {h1 == h2}")
```

Consider: if an attacker changed an IC50 value in the database, how would the system detect it?
